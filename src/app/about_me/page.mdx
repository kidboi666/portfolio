## 왜 웹 프론트엔드 였을까?.

웹은 바로 전달될 수 있는 힘이 있다고 생각합니다. 앱스토어 심사 없이 바로 배포할 수 있고, 브라우저만 있으면 누구나 접근할 수 있으며, 검색을 통해 자연스럽게 유입될 수도 있습니다.

React -> Next.js -> React Native -> Flutter 까지 경험하며 **"사용자에게 닿는 속도 + 접근성 + 확장성"** 이 세 가지가 웹 프론트엔드의 매력이라는 걸 확실히 느꼈습니다.

개인 서비스 OneLog를 개발하며 기존 계층형 구조를 Feature-Sliced Design 으로 전환했고, 이 과정에서 '돌아가는 코드'가 아니라 '지속 가능한 코드'를 만드는 법을 배웠습니다.

---

## 나를 표현하는 다섯 가지.

### 유튜브 영상 하나로 시작해서 앱스토어 배포까지 간 사람

"항상 학습 진도가 빠르고 오래 집중하는 사람."

이 문장은 부트캠프 동료에게 실제로 들은 피드백입니다. 2024년 유튜브 영상을 계기로 시작한 개발이 6개월 뒤엔 3개의 팀 프로젝트, 그리고 실제 서비스 운영까지 이어졌습니다.

- 팀 프로젝트 3개 **평균 기여도 **40~46%** (Github 커밋 기준)
- 개념을 배우면 바로 프로젝트에 적용하는 스타일
- "아이디어가 바로 화면으로 나타나는 과정"을 정말 좋아합니다.

저는 속도만 빠른 게 아니라, **실제로 끝까지 완주하는 타입**입니다.

---

### 6개월, 1인 개발, 32000줄 작성

*"MoodLog 감정 기록 앱 - 기획부터 출시까지 혼자 완주"*

처음엔 React Native로 시작했지만, 높은 러닝 커브와 빈번한 버그 대응에 어려움을 겪었습니다. 더 나은 개발 경험을 위해 Flutter로 전환을 결정했고, 2개월 만에 32,000줄을 재작성 했습니다.

- 코드 재작성 : 32,000줄 / 2개월
- Google Play · App Store 정식 출시
- 출시 후 4개월간 29회 업데이트
- 개인사업자 등록, QA, 심사 대응까지 직접 수행

이 과정에서 배운 가장 큰 한 줄은 이것입니다.

**"코드는 기능을 만드는 것이 아니라, 사람에게 가치를 전달하는 일이다."**

---

### 코드가 커질수록 설계에 눈이 가는 사람

개인 프로젝트 OneLog 개발 중, 컴포넌트 의존성 꼬임 -> 유지보수 어려움 -> 확장성 저하 문제를 맞았습니다.

그래서 Feature-Sliced Design을 도입했습니다.

- 6개 레이어 (app, pages, widgets, features, entities, shared) 로 구조 재편
- 도메인 로직 분리 / 순환 참조 차단
- 마이그레이션 과정에서 Claude Code 적극 활용

이 경험으로 **'돌아가기만 하는 코드'와 '지속 가능한 코드'의 차이**를 몸으로 이해했습니다.

---

### 클릭 한 번도 허투루 보지 않는 사람

프론트엔드의 본질은 '사용자의 관점' 이라고 믿습니다.

- 팀원들이 API 응답을 보기 어려워할 때 -> 관리자 페이지 자발적으로 제작
- OneLog 에서 디바이스 크기별 메뉴 위치 수십 번 테스트
- TanStack Query Optimistic Update 적용 -> 즉각 반응하는 UI 경험 제공

작은 불편이 쌓이면 사용자는 이탈합니다.

그래서 저는 작은 불편함을 예민하게 감지하는 사람입니다.

---

### 막히면 멈추는 대신, 끝까지 해결하는 체질

5년간 건설 현장에서 익힌 **계획 -> 실행 -> 검증 -> 책임** 이 루틴은 개발에서도 그대로 이어졌습니다.

Drizzle ORM 마이그레이션, FSD 전환, Server Component 최적화 등 쉽지 않은 문제를 만날수록 오히려 흥미가 올라가는 타입입니다.
